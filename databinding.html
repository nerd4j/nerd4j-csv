NERD4J-CSV

So far this is the quickest, easiest and most configurable open source Java framework for CSV manipulation.

Map CSV into different data models

The easiest and most common way is to map a CSV record into a data model bean, but sometimes we may have different needs. For example we may need our model to be a java.util.Map or a List of fields.

To map a CSV record into a java bean the nerd4j-csv framework uses reflection and therefore if you need performance it may not be the best choice. Luckily nerd4j provides the possibility to map into different models. The built in CSVToModelBinder are:

  • CSVToBeanBinder: the one used behind the scenes in the previous chapters. If you use annotations this binder is automatically configured.
  • CSVToMapBinder: Maps a CSV record into a java.util.Map where the column name (or index) is used as key and the column field as value.
  • CSVToArrayBinder: Maps the CSV record into an array of Object, in this case the information about the column name is lost and only the column indexes are used.

To use a different model binder we just need to change the XML configuration as follows:

<?xml version="1.0" encoding="UTF-8"?>
<csv:configuration xmlns:csv="http://www.nerd4j.org/csv"  
                   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:schemaLocation="http://www.nerd4j.org/csv http://www.nerd4j.org/csv ">

  <!-- CSV-READER CONFIGURATION -->
  <csv:reader name="modelReader">

    <csv:model-binder type="map" />

    <csv:columns>

      <csv:column name="user-name" mapping="name" />
      <csv:column name="password"  mapping="password" optional="true" />
      <csv:column name="user-age"  mapping="age" processor-ref="parseInteger" />

      <csv:column name="user-type" mapping="type">
        <csv:processor>
          <csv:converter type="parseEnum" enum-type="full.path.to.UserType" />
        </csv:processor>
      </csv:column>

      <csv:column name="registration" mapping="registration">
        <csv:processor>
          <csv:converter type="parseDate" pattern="dd-MM-yyyy" />
        </csv:processor>
      </csv:column>

    </csv:columns>
  </csv:reader>  

</csv:configuration>

As you can see you just need to change the csv:model-binder tag and everything works fine. Of course if you choose to use the model-binder type array all mapping attributes must contain a valid index in the target array.

Its also possible to register a custom binder. To do so as first you need to provide an implementation of the org.nerd4j.csv.reader.binding.CSVToModelBinder interface that performs the binding from the raw data into the needed data model, or implement the org.nerd4j.csv.writer.binding.ModelToCSVBinder that provides the data (in you custom data model) to the CSV writer.

As second step you need to provide an implementation of the org.nerd4j.csv.reader.binding.CSVToModelBinderFactory interface which is able to produce a org.nerd4j.csv.reader.binding.CSVToModelBinder according to the given configuration. There is an abstract implementation that provides most of the work out of the box.

Finally you need to provide an implementation of the CSVRegistryEntryProvider to be able to register your custom binder in the registry as we have seen for the registration of custom CSVFieldProcessors.

Once you have the CSVRegistryEntryProvider the entry in the XML configuration is the same used to register our custom CSVFieldProcessor in the previous chapter:

<csv:register>
...
  <csv:types>
    <csv:csv-to-model type-name="myCsvToModelBinder" provider-class="full.path.to.MyCsvToModelBinderFactory"/>
  </csv:types>
...
</csv:register>

This way you are able to map your CSV file into any data model you need. Similarly you can map any data model into a CSV format by providing the right ModelToCSVBinder and the related factory and CSVRegistryEntryProvider.
